# Design: Implement Subagent Execution Loop

## Architecture

The subagent execution loop is centered around `TaskExecutor`, which coordinates the `ToolCallingLoop` with a specific set of tools and context.

### 1. Scope Firewall

To ensure safety, a "Scope Firewall" will be implemented as a decorator pattern wrapping individual `ITool` instances. The firewall intercepts all tool calls and validates that any file paths provided as arguments are within the `AllowedFileScope` of the current task.

**Design Details:**
- The firewall applies to **both read and write operations** to prevent information leakage.
- Path normalization will use `Path.GetFullPath()` to handle relative paths, symlinks, and case sensitivity.
- The firewall will be applied at tool resolution time via `ScopedToolRegistry`.

### 2. Verification Tools

New tools will be added to the registry:

**`run_build` Tool:**
- Executes `dotnet build` on the solution file specified in the task context.
- Returns structured output: `{ success: bool, logs: string, duration: TimeSpan, errors: string[] }`.
- Enforces a 5-minute timeout to prevent runaway builds.
- Scope firewall applies: build output paths must be within allowed scope.

**`run_test` Tool:**
- Executes `dotnet test` with `--logger:trx` to generate structured test results.
- Parses `.trx` files to extract: test count, pass/fail counts, failure details, and duration.
- Returns structured output: `{ success: bool, totalTests: int, passed: int, failed: int, failures: FailureDetail[], logs: string }`.
- Enforces a 10-minute timeout.
- Test results are automatically captured as UAT evidence artifacts.

### 3. Evidence Capture

The `TaskExecutor` will be responsible for aggregating evidence into the run's evidence directory:

**`tool-calls.ndjson`:**
- Each line is a JSON object representing one tool call and its result.
- Format: `{ timestamp: ISO8601, toolName: string, input: object, output: object, duration: TimeSpan, success: bool, error?: string }`.
- Captured via `IToolCallingEventEmitter` during loop execution.

**`changes.patch`:**
- Unified diff format (RFC 3881) of all modified files.
- Generated by comparing the initial state (git HEAD or baseline) with the final state after execution.
- Includes file headers with old/new paths and line counts.
- Deterministic: same changes always produce identical patch content.

**`execution-summary.json`:**
- Structured summary of the entire execution.
- Contains: task ID, start/end timestamps, total iterations, files modified, tool calls made, build/test results, completion status.
- Example:
  ```json
  {
    "taskId": "...",
    "startTime": "2026-02-19T...",
    "endTime": "2026-02-19T...",
    "iterations": 3,
    "filesModified": ["src/Services/ProjectService.cs"],
    "toolCallsCount": 5,
    "buildResult": { "success": true, "duration": "00:00:15" },
    "testResult": { "success": true, "passed": 42, "failed": 0 },
    "completionStatus": "success",
    "deterministicHash": "sha256:..."
  }
  ```

**`deterministic-hash`:**
- SHA256 hash of the concatenated content: `tool-calls.ndjson` + `changes.patch` + `execution-summary.json`.
- Provides integrity verification and deduplication capability.
- Deterministic: same execution always produces the same hash.

## Data Flow

1.  `TaskExecutor` receives `TaskExecutionRequest`.
2.  `TaskExecutor` starts a new Run via `IRunLifecycleManager`.
3.  `TaskExecutor` initializes `ToolCallingLoop` with a `ScopedToolRegistry`.
4.  `ToolCallingLoop` executes iteratively, calling tools.
5.  `ScopedToolRegistry` enforces the firewall.
6.  Upon completion, `TaskExecutor` collects diffs and tool logs.
7.  `TaskExecutor` writes final evidence and updates the task status.
