# Proposal: Implement Real Chat Responder

## Change ID
`implement-chat-responder`

## Problem Statement

The orchestrator's chat response path is currently **stubbed with hardcoded responses** rather than integrated with the LLM runtime. When the `InputClassifier` correctly identifies input as `SideEffect.None` (chat intent), the system returns placeholder strings instead of genuine assistant dialogue.

### Current Behavior
- `ChatResponder.Respond()` returns: `"Hey — what do you want to work on?"` (fixed string)
- `ResponderHandler.HandleAsync()` returns: `"This is a conversational response."` (fixed string)

### Desired Behavior
- Chat responses are generated by an LLM using the configured provider (`ILlmProvider`)
- Responses include workspace awareness (existing specs, current state, available commands)
- Responses can be streamed to the UI for real-time dialogue feel
- The gating engine's decisions are visible to the user through conversational explanations

## Scope

This change implements the real chat responder path end-to-end:

1. **Core Chat Service** - LLM-backed response generation via `ILlmProvider`
2. **Context Assembly** - Workspace state and spec awareness for grounded responses
3. **Streaming Integration** - Optional SSE streaming for chat tokens
4. **Tool-Augmented Chat** - Read-only tools available during chat (file lookup, status check)

## Out of Scope (Future Work)

- Multi-turn conversation history persistence
- Tool-augmented chat with write operations (requires confirmation gates)
- Negotiation flows ("I think you want to plan phase PH-0002; should I proceed?")
- Custom personality/prompt template configuration
