# agents-subagent-execution Specification

## Purpose

Defines the requirements for the subagent execution loop, including tool calling, scope enforcement, and evidence capture.

- **Lives in:** `nirmata.Agents/*`
- **Owns:** Subagent execution loop logic, tool coordination, and evidence aggregation.
- **Does not own:** LLM provider implementations and core AOS state management.

## ADDED Requirements

### Requirement: Subagent execution loop iteratively calls tools until completion
The system SHALL provide an iterative execution loop that sends the task context to the LLM, executes requested tools, and continues until the LLM indicates the task is complete or a limit is reached.

The loop MUST:
- Use `IToolCallingLoop` to manage the conversation and tool execution.
- Terminate when the LLM provides a final response without further tool calls.
- Enforce maximum iteration and token limits to prevent runaway loops.

#### Scenario: Subagent completes task in multiple iterations
- **GIVEN** a task requiring two file modifications.
- **WHEN** the execution loop starts.
- **THEN** it iterates twice to call `write_file` for each modification, and then terminates on the third iteration with a completion summary.

### Requirement: Scope firewall enforces strict file access at the tool layer
The system SHALL implement a "Scope Firewall" that intercepts all file-related tool calls (both read and write) and rejects those attempting to access paths outside the `AllowedFileScope`.

The firewall MUST:
- Be applied at the tool execution level via a decorator pattern (`ScopedTool` wrapping `ITool`).
- Support both directory-level and file-specific scopes.
- Normalize paths using `Path.GetFullPath()` to handle relative paths, symlinks, and case sensitivity.
- Apply to **both read and write operations** to prevent information leakage.
- Return a deterministic `ScopeViolationException` if a scope violation is detected.
- Prevent any filesystem mutation before validation.

#### Scenario: Tool call rejected by scope firewall
- **GIVEN** an `AllowedFileScope` of `["src/services/"]`.
- **WHEN** a tool is called to write to `src/api/Program.cs`.
- **THEN** the firewall rejects the call with a `ScopeViolationException` before any filesystem mutation occurs.

#### Scenario: Symlink traversal is blocked
- **GIVEN** an `AllowedFileScope` of `["src/services/"]` and a symlink at `src/services/link -> ../api/`.
- **WHEN** a tool is called to write to `src/services/link/Program.cs`.
- **THEN** the firewall normalizes the path and rejects it as out-of-scope.

### Requirement: Execution loop provides verification tools
The system SHALL provide tools for building and testing the codebase within the subagent loop.

Tools MUST include:

**`run_build` Tool:**
- Executes `dotnet build` on the solution file specified in the task context.
- Returns structured output: `{ success: bool, logs: string, duration: TimeSpan, errors: string[] }`.
- Enforces a 5-minute timeout to prevent runaway builds.
- Scope firewall applies: build output paths must be within allowed scope.

**`run_test` Tool:**
- Executes `dotnet test` with `--logger:trx` to generate structured test results.
- Parses `.trx` files to extract: test count, pass/fail counts, failure details, and duration.
- Returns structured output: `{ success: bool, totalTests: int, passed: int, failed: int, failures: FailureDetail[], logs: string }`.
- Enforces a 10-minute timeout to prevent runaway test suites.
- Test results are automatically captured as UAT evidence artifacts.

#### Scenario: Subagent verifies changes with tests
- **GIVEN** a task to fix a bug.
- **WHEN** the subagent finishes modifying code and calls `run_test`.
- **THEN** the loop provides the test results back to the subagent to decide the next step.

#### Scenario: Build tool enforces timeout
- **GIVEN** a build that would take longer than 5 minutes.
- **WHEN** the `run_build` tool is called.
- **THEN** the tool terminates after 5 minutes and returns `{ success: false, logs: "...", errors: ["Build timeout after 5 minutes"] }`.

### Requirement: Task execution produces comprehensive evidence
The system SHALL aggregate and persist all execution artifacts into the run's evidence directory.

Evidence MUST include:

**`tool-calls.ndjson`:**
- Each line is a JSON object representing one tool call and its result.
- Format: `{ timestamp: ISO8601, toolName: string, input: object, output: object, duration: TimeSpan, success: bool, error?: string }`.
- Captured via `IToolCallingEventEmitter` during loop execution.
- Deterministic: same tool calls always produce identical log entries.

**`changes.patch`:**
- Unified diff format (RFC 3881) of all modified files.
- Generated by comparing the initial state (git HEAD or baseline) with the final state after execution.
- Includes file headers with old/new paths and line counts.
- Deterministic: same changes always produce identical patch content.

**`execution-summary.json`:**
- Structured summary of the entire execution.
- MUST contain: taskId, startTime, endTime, iterations, filesModified, toolCallsCount, buildResult, testResult, completionStatus, deterministicHash.
- Example structure:
  ```json
  {
    "taskId": "task-123",
    "startTime": "2026-02-19T19:45:00Z",
    "endTime": "2026-02-19T19:45:30Z",
    "iterations": 3,
    "filesModified": ["src/Services/ProjectService.cs"],
    "toolCallsCount": 5,
    "buildResult": { "success": true, "duration": "00:00:15" },
    "testResult": { "success": true, "passed": 42, "failed": 0 },
    "completionStatus": "success",
    "deterministicHash": "sha256:abc123..."
  }
  ```

**`deterministic-hash`:**
- SHA256 hash of the concatenated content: `tool-calls.ndjson` + `changes.patch` + `execution-summary.json`.
- Provides integrity verification and deduplication capability.
- Deterministic: same execution always produces the same hash.

#### Scenario: Execution evidence is persisted for audit
- **GIVEN** a completed task execution.
- **WHEN** inspecting the run folder.
- **THEN** it contains `tool-calls.ndjson`, `changes.patch`, `execution-summary.json`, and `deterministic-hash`.

#### Scenario: Evidence is deterministic
- **GIVEN** two identical task executions with the same inputs.
- **WHEN** comparing their evidence artifacts.
- **THEN** the `deterministic-hash` values are identical, and the patch files are byte-for-byte identical.

### Requirement: Loop handles completion and error states
The system SHALL properly handle task completion and error conditions.

The loop MUST:
- Detect natural completion when the LLM provides a response without tool calls.
- Enforce a maximum iteration limit (e.g., 20 iterations) to prevent runaway loops.
- Handle `ScopeViolationException` by capturing the error in evidence and terminating with error status.
- Handle tool execution timeouts by capturing the timeout in evidence and allowing the LLM to decide next steps.
- Handle LLM provider errors by propagating with context and capturing in evidence.

#### Scenario: Loop terminates on natural completion
- **GIVEN** a task that requires one file modification.
- **WHEN** the subagent modifies the file and the LLM responds without requesting further tool calls.
- **THEN** the loop terminates with `completionStatus: "success"` and persists all evidence.

#### Scenario: Loop terminates on max iterations
- **GIVEN** a task execution that reaches 20 iterations without natural completion.
- **WHEN** the 20th iteration completes.
- **THEN** the loop terminates with `completionStatus: "max_iterations_reached"` and persists evidence.

#### Scenario: Scope violation terminates loop
- **GIVEN** a task with `AllowedFileScope: ["src/services/"]`.
- **WHEN** a tool call attempts to write to `src/api/Program.cs`.
- **THEN** the loop terminates with `completionStatus: "scope_violation"` and evidence includes the violation error.
