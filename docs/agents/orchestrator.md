## Orchestrator Agent

### Responsibilities

- Own the control loop: classify → gate → dispatch → validate → persist → next.
- Enforce spec-first sequencing (no execute without plan; no plan without roadmap/spec).
- Load and maintain canonical truth across:
  - **Intended truth**: `.aos/spec/**`
  - **Operational truth**: `.aos/state/state.json` + `.aos/state/events.ndjson`
  - **Provable truth**: `.aos/evidence/**`
- Assemble deterministic context packs: `.aos/context/packs/**`.
- Coordinate codebase intelligence refresh: `.aos/codebase/**` (+ cache).
- Validate all produced artifacts against schemas/contracts.
- Open/close runs and attach auditable evidence (RUN-*, TSK-* evidence).
- Handle failures via explicit transitions (issue → re-plan/fix → re-execute → re-verify).
- Maintain resumability (events + checkpoints) and safe concurrency (locks).

### Step Format (single run)

1. Receive input (CLI command or freeform) and classify it into a concrete workflow mode/command.
2. Ensure workspace exists (`aos init` if `.aos/` missing), then load current config (`aos config show`).
3. Load canonical artifacts needed for routing: `.aos/spec/**`, `.aos/state/**`, `.aos/evidence/**`, and `.aos/codebase/**` if relevant.
4. Run pre-flight validation gates: `aos validate schemas` then `aos validate spec/state/codebase/evidence` as applicable.
5. Decide the state-machine transition (gating rules):
   - If `spec/project.json` missing → **Interviewer**
   - Else if `spec/roadmap.json` missing → **Roadmapper**
   - Else if no `spec/tasks/TSK-*/plan.json` for target → **Planner**
   - Else → **Executor**
   - After execute → **Verifier**
   - If verify fails → **Fix Planner** → **Executor**
6. Start a run record: `aos run start [--task TSK-######]`, and record the triggering input in the run log.
7. Assemble the execution packet (objective, constraints, references, output contract with exact file paths).
8. Build/refresh the relevant context pack: `aos pack build --task TSK-######` (or `--phase PH-####`).
9. Dispatch the selected specialist workflow with strict scope boundaries (allowed files/dirs to touch).
10. Collect outputs and validate contracts (required files exist + schema-valid JSON via `aos validate spec`).
11. Persist operational truth: append event to `.aos/state/events.ndjson`, update cursor in `.aos/state/state.json`, optionally checkpoint.
12. Persist provable truth: attach artifacts/logs to `.aos/evidence/runs/RUN-*` and update `.aos/evidence/task-evidence/TSK-*/latest.json`.
13. Finish the run: `aos run finish --status pass|fail` and record next transition.
14. Return status to user: what changed (exact paths), current cursor, and the next recommended `aos …` command.

### Summary

The Orchestrator is the control plane that keeps the system sequenced, validated, and resumable. It classifies each input, loads canonical truth from `.aos/spec/`, `.aos/state/`, and `.aos/evidence/`, enforces gating rules, builds context packs, dispatches the right specialist workflow, validates outputs against schemas, persists state/events/evidence, and returns a concise status plus the next action.

## Subagent Orchestrator

### Responsibilities

- Spawn a fresh subagent context per atomic plan step to prevent long-horizon context degradation.
- Enforce the invariant: each task plan contains at most **3 executable steps** — one step = one subagent dispatch (the "fresh context every time" execution model).
- Provide execution lifecycle control for each atomic step:
  - Mint a resumable execution identifier (RUN-* from `aos run start`)
  - Bind evidence, logs, and artifacts to that run folder under `.aos/evidence/runs/RUN-*`
- Assemble minimal, bounded context for the subagent via context packs: `.aos/context/packs/TSK-*.json` generated by `aos pack build --task … --budget …`
- Return a normalized execution result to the Orchestrator:
  - Status (pass/fail)
  - Verification outputs (logs/artifacts/UAT references)
  - Deviation/scope-expansion signals that must force replan (often with linked ISS-*)

### Step Format (single run)

1. Receive a dispatch from Orchestrator for a single atomic plan step (e.g., `TSK-######` Step 1 of 3) and its execution packet (objective/constraints/references/output contract).
2. Create a new execution id and open a run folder (`.aos/evidence/runs/RUN-*/`) bound to that one step; record the packet and allowed scope.
3. Build the minimal context pack for the step (`.aos/context/packs/...`) and spawn a fresh subagent with only that pack + the allowed-scope contract.
4. Execute the step and its required verification check, writing outputs only to the contracted locations.
5. Validate step outputs (schema + required files) and emit a normalized result record (status, verification evidence, scope-expansion flags) back to the Orchestrator; close the run.

### Summary

The Subagent Orchestrator is the execution isolator that keeps long-horizon work stable. It enforces the ≤ 3 steps per task plan rule, spins up a fresh subagent for each atomic step using a minimal context pack and strict file-scope contract, generates a resumable execution id, captures verification evidence inside an auditable RUN-* record, and returns a normalized pass/fail + scope-expansion signal to the Orchestrator so the system can either advance to the next step with a fresh subagent or force a re-plan.

## Context Engineer Agent

### Responsibilities

- Build the exact, deterministic context bundle required for the next run using canonical artifacts as the source of truth (not chat history).
- Translate mode (plan / execute / verify / fix / codebase) into a minimal artifact set pulled from `.aos/spec/**`, `.aos/state/**`, and optionally `.aos/codebase/**`.
- Enforce context-size budgets by trimming, prioritizing, and summarizing so downstream agent quality does not degrade.
- Persist context as a reusable context pack in `.aos/context/packs/**` (conforming to `context-pack.schema.json`) so runs are reproducible.
- Provide downstream agents a stable grounding packet with ordered references and explicit file paths to cite.

### Step Format (single run)

1. Identify the target mode and target scope (task `TSK-######` or phase `PH-####`).
2. Select the minimum canonical artifacts for that mode (always prefer `.aos/spec/**` + `.aos/state/state.json`, add `.aos/codebase/**` only when planning or brownfield changes require it).
3. Build the context pack deterministically with an explicit size budget using `aos pack build --task … --budget <n>` or `aos pack build --phase … --budget <n>`.
4. Validate the pack is well-formed and usable (schema-valid, ordered, and contains only referenced files that actually exist) and persist it to `.aos/context/packs/<TSK|PH>-*.json`.
5. Hand off the pack reference to the next agent run (planner/executor/verifier), using `aos pack show …` as the single canonical input surface.

### Summary

The Context Engineer is the packaging layer that makes execution reliable and repeatable. It takes a requested mode (plan/execute/verify/fix) and composes a deterministic, size-bounded context pack from the workspace's authoritative artifacts — primarily `.aos/spec/**` and `.aos/state/**`, with `.aos/codebase/**` added only when needed — then persists that pack under `.aos/context/packs/**` so downstream agents operate from stable files and explicit references rather than accumulated chat context.

## State Manager Agent

### Responsibilities

- Maintain the durable living memory of the workspace so the system is resumable and deterministic.
- Act as the operational source of truth for workflow position: current milestone/phase/task, status, and next action cursor.
- Record and normalize decisions (constraints, approvals, choices), blockers (impediments), and progression (cursor moves or branches).
- Prevent drift by ensuring downstream agents/commands read from persisted state, not implied chat context.
- Maintain an append-only event trail for auditability and replay.

### Step Format (single run)

1. Load the current operational truth from `.aos/state/state.json` and the latest event tail from `.aos/state/events.ndjson`.
2. Receive an update intent from Orchestrator (e.g., roadmap created, plan saved, step executed, verification passed/failed, fix requested, pause/resume, milestone transition).
3. Append a normalized event to `.aos/state/events.ndjson` capturing what changed and why (who/what/when + references to affected artifacts).
4. Apply the event to update the three core state domains: **decisions**, **blockers**, and **position** (cursor).
5. Persist immediately by writing the updated `.aos/state/state.json` (and optionally a checkpoint in `.aos/state/checkpoints/**` when the cursor crosses a milestone boundary).
6. Return the updated cursor + any gating signals (blocked, needs-input, needs-plan, ready-to-execute, needs-verify, needs-fix) to the Orchestrator.

### Summary

The State Manager is the persistence backbone of the engine. It keeps an explicit, durable record of decisions, blockers, and current workflow position in `.aos/state/state.json`, with an append-only event history in `.aos/state/events.ndjson`. Every plan, execution, verification, and pause/resume operation reads and writes through this layer so the system can always resume from saved truth rather than relying on conversational context or implicit assumptions.
